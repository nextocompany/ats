02 · End-to-End Flow

เส้นทางผู้สมัคร (End-to-End)

เส้นทางสมบูรณ์ตั้งแต่ผู้สมัครยื่นใบสมัคร จนถึงการจ้างงานและ onboarding แบ่งเป็น 3 ช่วง: (ก) รับเข้า → (ข) คัดกรองอัตโนมัติด้วย AI เบื้องหลัง → (ค) ขั้นตอนที่ HR ทำเอง

ช่วง ก · รับใบสมัคร (Intake)
1 · ยื่นใบสมัคร
ผู้สมัครเลือกตำแหน่งใน Portal แล้ว apply (เลือกเรซูเม่ที่บันทึกไว้ หรืออัปโหลดใหม่ PDF/DOCX/รูป ≤10MB) + ยินยอม PDPA
→
2 · สร้าง Application
API บันทึกใบสมัคร สถานะเริ่มต้น pending และอัปโหลดไฟล์เข้า Blob
→
3 · เข้าคิว
API enqueue งาน application:process เข้า Redis (asynq) แล้วตอบผู้สมัครทันที
ช่องทางรับเข้าอื่นที่ลงสู่คิวเดียวกัน:
Portal apply
Bulk upload (HR)
Webhook ภายนอก
PeopleSoft pull
ช่วง ข · คัดกรองอัตโนมัติเบื้องหลัง (Worker Pipeline — intake → scored)
1
OCR — อ่านเรซูเม่
ดึงไฟล์จาก Blob → Azure Document Intelligence อ่านเป็นข้อความ · ถ้าความมั่นใจ < 0.7 ตั้งธงให้ตรวจมือ (ไม่ล้มงาน)
↓
2
Parse — แยกข้อมูลด้วย LLM
Azure OpenAI แปลงข้อความเป็นโปรไฟล์มีโครงสร้าง (ชื่อ ทักษะ ประสบการณ์ ฯลฯ)
→
⚠ ไม่ใช่เรซูเม่
→ สถานะ invalid_resume แจ้งผู้สมัครให้อัปโหลดใหม่ · จบงาน (ไม่ retry)
⚠ ชื่อไม่ตรงบัญชี
(เฉพาะ portal) → name_mismatch แจ้งผู้สมัคร · จบงาน
↓
3
Dedup — ตรวจผู้สมัครซ้ำ
เทียบชื่อ/เบอร์/อีเมล/บัตรปชช. หาผู้สมัครตัวจริง (canonical) · ถ้าซ้ำ ย้ายใบสมัครไปรวม
→
3b · สร้างบัญชี Portal เงียบๆ
ผูกบัญชี career-portal ให้ canonical อัตโนมัติ (ไม่มีการแจ้ง)
↓
4
Score — ให้คะแนนความเหมาะสม
โหลด Master JD → คะแนนทำเล (LocationScore) + คะแนนรวมแบบ rules + AI ถ่วงน้ำหนักต่อตำแหน่ง (ประสบการณ์/ทักษะ/วุฒิ/ภาษา/ทำเล)
↓
5
Must-have Gate — เกณฑ์บังคับ
ถ้าขาดคุณสมบัติบังคับ → ตัดออก
→
✕ ไม่ผ่านเกณฑ์
→ สถานะ rejected แจ้งผู้สมัคร · จบงาน
↓
6
Branch Assignment — จัดสาขา / talent pool
map จังหวัด→subregion→สาขา · บันทึกคะแนน · ตั้งสถานะ scored (= คัดกรองผ่าน) → แจ้งผู้สมัคร
↓
7
แจ้ง HR + index ค้นหา (best-effort)
อีเมล/Teams แจ้งทีมสำหรับผู้ที่ถูกจัดเข้าสาขา · re-index ผู้สมัคร canonical เข้า AI Search
ขั้นตอน pipeline (อัตโนมัติ)
สาขาที่ตัดออก/แจ้งเตือน
คิวงาน
หากเกิด error ชั่วคราว → สถานะ failed และ asynq จะ retry
ช่วง ค · ขั้นตอนที่ HR ทำเอง (Synchronous — ทำใน API ไม่ผ่าน Worker)
8 · สัมภาษณ์ AI
ผู้สมัครคุยกับ AI แบบ adaptive ผ่าน token
9 · วิเคราะห์ Fit
AI ประเมินความเหมาะข้ามตำแหน่ง
10 · Shortlist
SGM สาขาเลือก Top-5
11 · นัดสัมภาษณ์
สร้างนัด/Teams ผ่าน Graph · HR เป็น attendee
12 · บันทึกผล
Scorecard TA + LM แบบมีโครงสร้าง
13 · อนุมัติหลายระดับ
L1→L4: Staff→HRM→SGM→Regional
14 · ออกข้อเสนอ
ร่าง + ส่ง offer · ผู้สมัครตอบรับ/ปฏิเสธ + จดหมาย PDF
15 · Onboarding
เอกสารรับเข้า → จ้างงาน
▼
🏢 Sync กลับ PeopleSoft HCM
เมื่อ HR ตั้งสถานะ hired หรือผู้สมัครตอบรับข้อเสนอ → push ข้อมูลพนักงานใหม่ออกไปยัง PeopleSoft (best-effort)
Status Machine — วงจรสถานะใบสมัคร
pending→ parsed→ scored (คัดกรองผ่าน)
↓
ai_interview→ ai_interviewed→ shortlisted→ interview→ interviewed
↓
pending_approval→ offer→ hired ✓
สถานะแยกออกจากเส้นทางหลัก: invalid_resume name_mismatch rejected failed (retry)
คำอธิบาย

เส้นแบ่งสำคัญคือสถานะ scored — ทุกอย่างก่อนหน้านี้ทำอัตโนมัติใน Worker เบื้องหลัง หลังจากนั้นทุก transition เป็นการกระทำของ HR ผ่าน API โดยตรง (ไม่ผ่านคิว)

เกตปลอดภัยตั้งแต่ต้นทาง — ไฟล์ที่ไม่ใช่เรซูเม่ ชื่อไม่ตรง หรือขาดคุณสมบัติบังคับ จะถูกแยกออกตั้งแต่ต้น พร้อมแจ้งผู้สมัครอย่างสุภาพ และไม่ retry (กันการวนซ้ำไม่จำเป็น)

การแจ้งเตือนทุกสถานะ — แต่ละการเปลี่ยนสถานะที่สำคัญจะแจ้งผู้สมัครผ่าน LINE/อีเมล พร้อม deep-link กลับมาที่หน้าสถานะ (/status?token=) หรือหน้า offer

หมายเหตุ end-state — ในโค้ดสถานะ hired ถูกระบุว่าเป็น legacy terminal และ offer เป็นสถานะปลายทางอนาคต การ label ปลายทางกำลังอยู่ในช่วงเปลี่ยนผ่าน
