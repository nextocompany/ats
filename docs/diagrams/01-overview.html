ภาพรวมระบบ

มองระบบ ATS จากภายนอกเข้ามา ใครคือผู้ใช้ ระบบแกนกลางมีหน้าที่อะไร และพึ่งพาบริการภายนอกใดบ้าง เส้นทางทั้งหมดวิ่งผ่าน Go API หนึ่งเดียวที่เป็นประตูสู่ระบบ

ผู้ใช้งาน / Actors
👤 ผู้สมัครงาน
Candidate · ใช้ Career Portal เพื่อค้นหางาน สมัคร ติดตามสถานะ และตอบรับข้อเสนอ
🧑‍💼 ทีม HR / สรรหา
HR staff/manager, SGM (สาขา) · ใช้ HR Dashboard คัดกรอง สัมภาษณ์ อนุมัติ ออกข้อเสนอ
📊 ผู้บริหาร
Regional/Operation Director, Auditor · ดู Executive overview, รายงาน, การอนุมัติระดับสูง
🏢 PeopleSoft HCM
ระบบ HR กลางองค์กร · ส่งใบสมัคร/ตำแหน่งเข้ามา และรับข้อมูลพนักงานใหม่กลับเมื่อจ้าง

⚙️ Go API (Fiber) — ประตูหลักของระบบ
จุดเดียวที่รับ traffic จากผู้ใช้ทั้งหมด · ทำงาน HR แบบ synchronous (สัมภาษณ์ AI, fit, shortlist, อนุมัติ, ออกข้อเสนอ) และ โยนงานหนัก (อ่าน/คัดกรองเรซูเม่) เข้าคิว ให้ Worker ประมวลผลเบื้องหลัง
🔧 Worker
ประมวลผล pipeline คัดกรองเรซูเม่แบบ async + งานกวาดข้อมูลตามรอบ
⏰ Scheduler
ตั้งเวลา cron — ส่งออกรายงาน, กวาด retention, re-engage talent pool, SLA อนุมัติ
🗄️ ชั้นข้อมูล
PostgreSQL · Redis (คิว) · Blob (เรซูเม่) · AI Search (ค้นหา)

คำอธิบาย

ทำไม API เป็นประตูเดียว — ทั้ง Career Portal และ HR Dashboard เป็น Next.js ที่เรียก Go API โดยตรงผ่าน HTTPS (CORS อนุญาตเฉพาะสอง origin นี้) API เป็น Container App เพียงตัวเดียวที่เปิด public ingress สำหรับ business logic ส่วน Worker/Scheduler ไม่รับ traffic จากภายนอกเลย

งานหนักแยกออกจาก request — การอ่านและคัดกรองเรซูเม่ใช้เวลาและเรียก AI หลายครั้ง จึงไม่ทำใน request ตรงๆ แต่ API จะ enqueue งานเข้า Redis แล้ว Worker ดึงไปทำเบื้องหลัง ผู้สมัครได้รับการตอบกลับทันที และติดตามสถานะภายหลังได้

หลายช่องทางรับสมัคร แต่ pipeline เดียว — ไม่ว่าจะมาจาก portal, อัปโหลดเป็นชุด, ฟอร์มภายนอก หรือ PeopleSoft ทุกใบสมัครจะถูกโยนเข้า pipeline คัดกรองตัวเดียวกัน ทำให้ผลลัพธ์ (การให้คะแนน การจัดสาขา) สม่ำเสมอ

บริการภายนอกสลับ mock/real ได้ — ทุกการเชื่อมต่อภายนอกมีตัวสลับ *_PROVIDER โดยปริยายเป็น mock เพื่อให้พัฒนา/ทดสอบได้โดยไม่ต้องมีของจริง แล้วค่อยเปิด real ใน production (ดูแผนผัง 06)
