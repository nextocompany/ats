## 1. AI Parsing CV — real parsing of 20–30 CVs
**Screen:** `/applications/bulk` (left nav → "Bulk upload"). Backend `POST /api/v1/applications/bulk-intake`
→ async pipeline (OCR → parse).
**Steps**
1. Log in (super_admin) → left nav **Bulk upload** → `/applications/bulk`.
2. Pick a **Position** from the dropdown.
3. Select **20–30 real CV files** (pdf/docx/png/jpg, ≤10MB each, ≤50 files).
4. Click **อัปโหลด (N)**.
5. Result panel shows **สำเร็จ N / ล้มเหลว M** with a per-file list; click **เปิด** on a few to open
   `/applications/{id}`.
6. Wait ~1–3 min (pipeline is async), refresh the **Inbox** (`/applications`).
**Expected:** each upload becomes an application; after the pipeline runs, the placeholder filename is
replaced by the **real parsed candidate name** in the inbox, and the candidate's phone/email are populated
(visible on `/candidates/{id}`). Failures (corrupt/blank files) are listed with a reason and don't block the
batch.
**Note:** single-CV upload is a candidate-portal flow, not in the HR dashboard — use bulk for HR UAT.

## 2. AI Scoring — match score vs JD
**Screen:** Inbox `/applications` (Fit column) + `/applications/{id}` (score hero in AiSummaryPanel).
**Steps**
1. After Item 1 completes, open `/applications`.
2. Read the **Fit** column (0–100 score badge) and the **Requirements** column (Meets/Missing must-haves);
   list is sorted best-fit first.
3. Open a high-scoring and a low-scoring candidate → on `/applications/{id}` read the **score hero**
   (color: ≥75 green / ≥50 amber / else red) and the **Passed/Failed must-have gate** line.
**Expected:** scores vary by CV↔JD fit; a candidate failing the must-have gate (below min education/
experience) is **auto-rejected** (status `rejected`) with score reflecting the gate. Score = Experience/30 +
Skills/20 (AI) + Education/10 + Language/10 + Location/20.

## 3. AI Summary — strengths / concerns
**Screen:** `/applications/{id}` → **AiSummaryPanel → ScoreBreakdown**.
**Steps**
1. Open a scored candidate at `/applications/{id}`.
2. Read **จุดแข็ง** (3 Thai bullets), **ข้อสังเกต** (red flags), and the 5 labelled dimension bars
   (ประสบการณ์/ทำเล/ทักษะ[AI]/การศึกษา/ภาษา).
**Expected:** strengths + concerns are specific to the CV (generated by GPT-4o), not generic. Fields come
from `ai_summary` / `ai_red_flags` / `ai_score_breakdown` (only on the detail view, not the list).

## 4. Semantic Search — find candidate
**Screen:** `/search`. Backend `GET /api/v1/candidates/search?q=` (Azure AI Search, Thai analyzer).
**Steps**
1. Open `/search` (search box autofocuses).
2. Type a **Thai name** or **province** from your uploaded set, or click a province chip
   (กรุงเทพมหานคร/เชียงใหม่/…). Input is debounced (~300ms) and written to `?q=`.
3. Read the results table: **Score · Candidate · Province · Status**; each row = the candidate's best-scoring
   application; click a row → `/candidates/{id}`.
**Expected:** matches by name/province with relevance ranking (Thai names tokenized by `th.microsoft`).
**Caveat:** results appear only after Item 1 has indexed candidates; the `/search` UI sends only the text
query (status/min-score filters exist in the API but aren't wired into this page).

## 5. Branch Assignment — assign to สาขา
**Screen:** Inbox `/applications` **Placement** column + `/applications/{id}` badge. Automatic at intake
(haversine nearest open-vacancy store). No manual reassign UI.
**Steps**
1. In `/applications`, read the **Placement** column: store name, or **Talent pool**, or province fallback.
2. Open a candidate → `/applications/{id}` → see the `store {id}` / `talent pool` badge.
3. Cross-check: a candidate in a province with an open vacancy (matching the position's store format) gets the
   **nearest store**; an unknown province or no open vacancy → **Talent pool**.
**Expected:** every screened candidate is assigned a store or talent pool automatically.
**Manual reassign (added 2026-06-19):** on `/applications/{id}` the **ReassignControl** lets an authorised
role override the auto-assignment - pick a different store or move to/from Talent pool. Backend
`PATCH /api/v1/applications/{id}/assignment` (role-gated; `SetAssignment` with `store_no` or `pool=true`).
Verify: change a candidate's store, refresh the inbox **Placement** column, confirm it reflects the override.

## 6. User Access — role / permission
**Screen:** `/login`, `/admin` (super_admin only). RBAC scope on every list/report.
**Steps**
1. As super_admin → `/admin` → **User management** → create an **hr_staff** user with `store_id = X` + a
   password (≥10 chars, letter+digit).
2. Log out → log in as that user (password path at `/login`).
3. Go to `/candidates` and `/applications` → **expect only store X's candidates** appear.
4. (Edge) create an hr_staff user with **no store** → expect an **empty** list.
5. Log back as super_admin → all stores visible. Confirm nav items differ per role (e.g. `/executive` only for
   super_admin/regional_director/auditor; `/admin` only super_admin).
**Roles:** super_admin, regional_director, auditor, operation_director, sgm, hr_manager, hr_staff. Scope:
all / subregion / store (store-scoped with no store = sees nothing, fail-closed).

**Dynamic RBAC (added 2026-06-20):** roles + the role→permission matrix + per-role scope are now **DB-driven
and editable** at `/admin` → **Roles & Permissions** (super_admin only): create custom roles, toggle the
permission matrix, set scope. super_admin is a hard code bypass (never lockable). Permissions are a fixed
code catalog (21 keys); `/users/me` returns the effective `permissions[]` + `scope`.

**Entra SSO role + scope (important):** for **Microsoft SSO** logins the role and scope come **only from the
token claims**, not the DB - the dashboard matrix editor binds role→permission but does **not** set an Entra
user's role. Set the role via the Azure **app role** assignment, and store/subregion via **directory-extension
claims**. Full setup (7 app roles + `store_id`/`subregion` extensions + `az rest` commands) is in
`docs/entra-app-roles-setup.md`. UAT order: test scope=all roles (super_admin/regional_director/auditor) first
- they need no extra claims - then store-scoped roles after the extensions are set. The **password-login** path
(create user at `/admin`) reads role/scope from the DB and needs no Azure setup, so it is the simpler path for
the scope test above.

## 7. Concurrent Load - concurrent usage
**Type:** ops runbook (`loadtest/README.md`). **Already run on prod 2026-06-20** (recorded baselines in the
README): a safe **read-only** test (`loadtest/readonly-load.js`: 20 VUs x 2 min = ~40k reqs @ ~334 req/s,
0% errors, /health p95 67ms, api autoscaled 1→3) and a bounded **write-path + pipeline drain**
(`loadtest/intake-batch.js`: 100 CVs @ 10 VUs intake p95 583ms 0% err; pipeline ~85 CV/min @
WORKER_CONCURRENCY=10 / 1 worker, **zero Azure 429**) run on the empty prod stack with a full 3-layer cleanup
(DB + blobs + AI Search index, incl. an orphan sweep for async indexing). No staging stack exists.
**Re-running the write-path test** requires the same cleanup; do it on empty prod or a throwaway stack only.
**Steps (operator)**
1. `k6 run -e TARGET=https://<api> -e POSITION_ID=<uuid> -e COOKIE="hr_auth=<session>"
   -e SAMPLE=<ABSOLUTE path to sample> -e CVS=100 -e VUS=10 loadtest/intake-batch.js`.
   (k6 `open()` resolves SAMPLE relative to the script dir - **must be an absolute path**.)
2. Thresholds `http_req_failed<1%`, `p95<2s`.
3. Measure pipeline drain: bulk-submit a known batch, poll `GET /api/v1/applications?status=scored|rejected`
   until counts plateau → CVs/min.
**Expected:** API stays <1% errors / p95<2s; pipeline drain rate is bounded by the **Azure OpenAI TPM +
DocIntel** tier (429s → retried, no data loss). Tune `WORKER_CONCURRENCY` (default 10). Report CVs/min +
429 rate to size the Azure tier. **Do not run against prod `hr_db`** (creates real rows; cleanup SQL in the
README, rows tagged `source_channel='loadtest'`).

## 8. Candidate Tracking — track status
**Screen:** Inbox status pill + `/applications/{id}` actions + JourneyPanel; portal `/status` for candidates.
**Steps**
1. `/applications` → **Status** column (pill) + status filter dropdown.
2. Open a candidate → `/applications/{id}` → use the **Next step** buttons (Shortlist → Interview… →
   Mark interview done → Reject…[reason required]); state machine enforces legal moves.
3. Scroll the right rail to **ไทม์ไลน์ผู้สมัคร (JourneyPanel)** → each status change shows `from → to` +
   timestamp.
4. Candidate side: open the career portal `/status`, paste the tracking code (the `status_token` from apply)
   → see `status / applied_at / position`.
**Expected:** status transitions follow the funnel (scored→…→offer/rejected), illegal moves are blocked,
and every change appears on the timeline. Candidate sees a minimal status view via the opaque token (no UUID).

## 9. Deduplication — detect duplicates
**Screen:** runs in the pipeline (Step 3); surfaced as a `dedup: {state}` badge on `/applications/{id}`
AiSummaryPanel.
**Steps**
1. Upload the **same CV twice** (or two CVs sharing id_card/phone/email) for the same position via
   `/applications/bulk`.
2. Open the second application → `/applications/{id}` → look for the **`dedup: auto_merged`** (or
   `pending_review`) badge.
3. Verify the duplicate is **excluded from `/search`** results (auto-merged shadow points to the canonical
   candidate).
**Expected:** id_card match → `auto_merged` (≥0.9, repointed to canonical); contact+name → `pending_review`
(0.7–0.9, kept separate for HR). **Known gap to check:** an auto-merged shadow candidate can still appear in
the plain `/candidates` list (the list query doesn't filter `is_duplicate_of`) even though its application is
repointed — note if this is acceptable.

## 10. Dashboard — HR dashboard
**Screen:** `/dashboard` (Overview) + the surrounding nav.
**Steps**
1. Log in → land on `/dashboard`.
2. Verify the **KPI band**, **Waiting by store**, **Open roles**, and the **screening pass-rate %**
   (passed/applied) render with real numbers (after Item 1).
3. Click a quick-action → deep-links to a filtered `/applications` (e.g. `?status=scored`, `?min_score=75`).
4. Spot-check the other surfaces visible to your role: `/applications`, `/candidates`, `/search`,
   `/analytics`, `/reports`.
**Expected:** numbers reflect the uploaded data; quick actions filter the inbox correctly.

## 11. Notification — Email / Teams
**Screen:** none (background). Email is **live** (ACS); **Teams is OFF** on prod (no webhook).
**Steps**
1. **HR email:** bulk-upload a CV that screens + scores + assigns a store (Item 1) → the store's HR
   recipients get **"ผู้สมัครใหม่ผ่านการคัดกรอง"**.
2. **Candidate email:** move a candidate to `shortlisted` / `interview` / `offer` / `hired` on
   `/applications/{id}` → if the candidate has an email, they get **"อัปเดตสถานะใบสมัคร"** (rejections are
   intentionally not emailed).
3. Check the actual inboxes (real ACS send on prod).
**Expected:** emails arrive for the triggers above. **Teams:** not testable on prod until
`TEAMS_WEBHOOK_URL` is set (then HR-facing messages also post to the channel).

## 12. Language — TH / EN
**Screen:** the TH/EN pill (LocaleSwitcher) in the dashboard chrome; cookie `NEXT_LOCALE`, default `th`.
**Steps**
1. Click the **EN** pill → page re-renders.
2. Verify the **translated** surfaces switch language: nav labels, `/shortlist`, `/approvals`,
   `/executive`, `/reports`, offer/letters/onboarding panels, `/login`.
3. Note the **still-hardcoded** surfaces (do not switch yet): `/dashboard`, `/applications` (inbox),
   `/candidates`, `/search`, `/analytics`, `/admin`, `/members` — these are the i18n follow-up.
4. Career portal: `/jobs` and a job detail **do** switch TH/EN (extracted 2026-06-19).
**Expected:** translated surfaces flip TH↔EN; hardcoded dashboard surfaces stay in their original language
(known partial coverage — bilingual is complete for the candidate-facing portal jobs flow, partial for the
internal HR dashboard).
